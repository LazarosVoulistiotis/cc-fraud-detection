# Week 20 — Report Snippet: Live Cloud Deployment, Monitoring Concept, and Architecture

In Week 20, the fraud detection API moved from a locally validated Docker runtime to a live cloud-hosted inference service through deployment on Google Cloud Run. A dedicated Google Cloud project was configured, the required Cloud Run, Cloud Build, and Artifact Registry APIs were enabled, and the repository was prepared in Cloud Shell for deployment. During the first deployment attempt, the build process revealed that the frozen model artifact and demo lookup CSV required by the serving container were not present in the cloned repository build context. This issue was resolved by uploading the missing runtime artifacts, creating a `.gcloudignore` file to control build inclusion, and switching to a more explicit two-step workflow: first building and pushing the container image with Cloud Build, and then deploying that image to Cloud Run.

The final deployment completed successfully and produced a public Cloud Run endpoint for the fraud detection API. The live service exposed the expected routes, including `/health`, `/metadata`, `/predict`, `/predict_by_id`, and `/docs`. Validation confirmed that the deployed service was operational, that the FastAPI Swagger interface loaded correctly, that the metadata endpoint exposed the frozen serving configuration, and that `/predict_by_id` returned a valid end-to-end model prediction using the locked XGBoost champion model and threshold policy. In parallel, a deployment architecture diagram was created to document the live serving path from client request to JSON response, and a lightweight monitoring and drift strategy was defined conceptually to cover latency, alert behaviour, precision/recall drift, feature distribution drift, and retraining triggers. As a result, Week 20 completed the transition from deployment-ready packaging to a production-inspired live inference system.
